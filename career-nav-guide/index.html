页面是怎么连起来的？

Next.js 用一种很聪明的方式管理页面——文件夹名就是网址。你只要看文件夹结构，就知道用户能访问哪些页面。

这叫「约定优于配置」

传统框架需要你手动写路由表（"网址 /quiz 对应这个文件"）。Next.js 反过来：文件放对了位置，路由自动就有了。这是现代 Web 框架最常见的设计哲学之一。

8 道题如何变成雷达图

情境判断题背后的评分算法——比「对或错」聪明得多

这不是考试，没有标准答案

传统的性格测试（比如 MBTI）是「选 A 得 1 分，选 B 得 0 分」。career-nav 用的是更精密的方法：每个选项在一条光谱上有位置。

比如「工作风格」这个维度，光谱的一端是「按部就班」，另一端是「灵活应变」。选不同选项，你就在光谱上落在不同位置——没有好坏，只有「适合哪种工作环境」的差异。

为什么不叫 MBTI？

因为 MBTI 是注册商标，还有学术争议。代码里有一条硬规定：「严禁出现 MBTI / 大五人格 / 霍兰德等专有名词，对外统一称职业偏好量表」。这是政府项目的合规要求。

每个选项都有一个「光谱位置」

在代码里，这个光谱位置叫 poleValue（极点值），是 0-100 的数字。0 表示偏左极，100 表示偏右极。

8 道题，每维 2 道题。你在每道题上选的选项，都有一个 poleValue。两道题取平均，就是你这个维度的最终得分。

同一道题，还能同时测 6 个能力？

除了 4 个性格维度，量表还会算出 6 个能力维度（用于职业定位的雷达图）。这是通过每个选项的「权重」来实现的。

评分算法核心 (lib/scoring.ts) 

for (const k of ABILITY_ORDER) {
  // 本题对能力 k 的理论最高分
  let maxW = 0;
  for (const opt of q.options) {
    if (opt.weights[k] > maxW) maxW = opt.weights[k];
  }
  maxPossible[k] += maxW;
  achieved[k] += selected.weights[k] ?? 0;
}
白话翻译

对每个能力维度（沟通、协作、执行…）都算一次

先找出「这道题里，这个能力的天花板分数是多少」

初始化天花板为 0

遍历这道题的所有选项

找最高权重（即「最可能体现这个能力的选项」的分数）

结束遍历

把天花板加进「理论最高」计数器

把用户实际选的选项权重加进「实际得分」计数器

最后：实际得分 ÷ 理论最高 × 100，就是 0-100 的能力分。这样不同题目数量的维度也能公平比较。

报告里的 6 个能力维度从哪来？

量表的每道题，每个选项都有 6 个维度的权重分配。8 道题答完，6 个累积器里就有了各维度的「已得分 / 理论最高」，最终换算成雷达图。

03

AI 是怎么写报告的

5 个模块，一条链，每个 AI 都知道前面的 AI 说了什么

报告不是一次 AI 调用生成的

你可能以为 AI 报告是这样的：把所有信息发给 AI，AI 一次吐出 5 个模块。

实际上，这个项目用了一种更聪明的方式——5 个模块按顺序生成，每个模块都知道前面模块的结论。这叫「Context Threading」（上下文穿针）。

1
① 总评

输入：量表分 + 访谈 Q1/Q2 → 输出：性格定位 + 四维雷达

2
② 优势发现

输入：能力分 + ①的性格定位 → 输出：3条优势 + 2条成长建议

3
③ 职业定位

输入：评分 + ①的性格 + ②的优势 → 输出：首选/次选岗位

4
④ 简历快诊

输入：简历文字 + ①的性格 → 输出：简历问题 + 改写示例

5
⑤ 行动建议

输入：所有评分 + ③的推荐岗位 + ①② → 输出：3个最重要的行动

💡
为什么不让 AI 一次生成全部？

因为如果 AI 一次要生成 5 个模块，它很容易在后面「忘掉」前面说的结论，导致总评说「你适合销售」，行动建议却说「去考程序员证书」。顺序生成 + context threading 让报告逻辑自洽。

AI 模块之间是怎么「对话」的？

每个后续模块都会拿到前面模块的关键结论。来看职业定位模块是怎么用前面两个模块输出的：

①总评
总评模块 (generateOverview)

生成完毕。性格定位：「稳健型执行者」，性格标签：踏实、专注、责任心强。

②优势
优势模块 (generateStrength)

收到总评结论。我的描述语气会和「稳健型执行者」保持一致。优势：细节把控能力强、流程执行扎实、团队协作稳定。

③定位
定位模块 (generatePositioning)

我拿到了：性格=「稳健型执行者」，主要优势=「细节把控、流程执行、团队协作」。基于这些，推荐首选岗位：行政主管；次选：项目协调专员。

⑤建议
行动建议 (generateAdvice)

好的，推荐岗位是「行政主管」和「项目协调」——那行动建议 #1 就是：本周内整理并精简简历中的行政管理经历，突出跨部门协调案例。

AI 返回的 JSON 坏掉了怎么办？

JSON 是代码里传递数据的标准格式。问题是：AI 有时候会输出「不规范」的 JSON——带中文标点、多一个逗号、或者没闭合括号。

代码里有一个叫 tryFixAndParse 的「JSON 急救函数」，专门修复这些问题：

「→"

中文引号

把 「」 替换成 "" ——AI 中文输出时常夹带中文标点

，→,

全角逗号

把中文逗号「，」替换成英文逗号「,」

,}

多余逗号

删除最后一个元素后多出来的逗号（这在标准 JSON 里是非法的）

{ →{}

未闭合括号

数花括号和方括号，发现没对齐就自动补全关闭符号

🔧
工程师怎么看待这个问题

不要假设 AI 总是返回完美的数据。防御性编程（defensive programming）就是假设外部数据可能是坏的，提前写好修复/兜底逻辑。这是生产级 AI 项目的标配。

4 道题，不是 4 道一样的题

访谈页面有 4 道问题，但它们的「含金量」完全不同——代码里有一个重要的设计决策隐藏在这里：

Q1

第 1 题

DeepSeek 根据你的简历动态生成。你的回答会进入报告的每一个模块。

Q2

第 2 题

DeepSeek 动态生成。你的回答同样进入报告。
答完这题，报告生成就在后台悄悄启动了！

Q3

第 3 题（缓冲）

从题库随机抽取。你的回答不进报告——这道题只是让你在等待时有事可做。

Q4

第 4 题（缓冲）

同上，从题库随机抽取，回答不进报告。答完这道题，报告生成通常也完成了。

💡
这是一个用户体验的小把戏

报告生成需要 40-100 秒（5 个 AI 模块顺序调用）。Q3/Q4 就是在「占用用户注意力」——让他们觉得「我还在答题」，而不是「我在等」。这叫 perceived performance（感知性能）优化。

访谈页有 14 种状态，为什么这么多？

访谈页的代码里有一个状态机管理整个交互流程。看起来很多，但每个状态都有其存在的理由：

🎬

初始化阶段

init → greeting → idle
播放欢迎语、等待用户准备好

🎤

录音阶段

requesting-mic → mic-granted → ready → recording
申请权限、拿到权限、等待按下、录音中

⚡

处理阶段

transcribing → preview
把音频发给 ASR 处理、展示识别文字

✅

完成与异常

done → /loading
error：网络问题、权限拒绝等

🔑
状态机的核心价值

没有状态机，你很难防止「按了两次录音按钮」「AI 还没说完用户就开始说」这类竞态条件问题。状态机像交通灯——确保每次只有一件事在发生。

代码里藏着的速度魔法——你的 AI 应用也能用

你以为报告是提交按钮按下才开始生成的？

不是的。代码里有三处「提前启动」，让用户感觉报告「一下子就出来了」。

这就像餐厅在你坐下来翻菜单的时候，厨房就已经在备料了——等你点完单，料已经准备好，上菜明显更快。

🚀
Layer 1：点「下一步」时就预触发

首页点提交 → startReportPrefetch(payload) 和 startQuizPrefetch(payload) 同时被调用 → 量表题目在后台预加载

🚀
Layer 2：Q2 答完立刻启动报告生成

用户答完访谈第 2 题 → startAfterQ2() 被调用 → AI 报告生成在后台开始，此时用户还在答 Q3/Q4

⏳
Layer 3：Loading 页消费已完成的 Promise

用户到达 loading 页时，报告可能已经生成一半甚至全部完成，直接消费结果，不需要重新等待

后台生成是怎么实现的？

Promise 是关键。代码在 Q2 答完时就发出了 AI 请求，并把这个 Promise 存在内存里。到了 loading 页，直接 await 这个已经在跑的 Promise 就行了。

代码翻译：Loading 页是怎么「取货」的

lib/report-client.ts — consumeAll 

const bgPromise = consumeBgGeneratePromise(
  formData, quizAnswers, interviewQ1Q2
);
if (bgPromise !== null) {
  sections = (await bgPromise) as GenerateSections;
}
if (!sections) {
  sections = await callGenerateWithRetry(callPayload, 1);
}
白话翻译

去「bg-runner 仓库」取之前存的那个 Promise（报告生成任务）

把用户数据作为「取货码」，确认是同一次会话的 Promise

取货结束

如果真的取到了 Promise（后台任务在跑或已完成）

等待它完成，拿到报告数据（可能只等很短时间）

结束

如果没取到 Promise（缓存过期或跳过了后台预热）

现场发起报告生成请求，最多重试 1 次（这是最慢的路径）

结束

再看几个你代码里的「工程师思维」

除了速度优化，代码里还有一些值得学习的设计决策：

🛡️

Prompt 注入防御

简历内容用 <resume> 标签包裹，防止用户在简历里写「忽略上述指令」来操控 AI 输出。

🎭

Mock 测试模式

设置 E2E_MOCK_MODE=true，所有 AI 调用返回预设数据——不花 API 费用就能测试完整流程。

🔑

幂等 UUID

每次填表生成唯一 ID，防止用户刷新报告页导致数据库出现重复记录。

⚡

内容校验器

AI 生成的每个字段都有 validator 检查——空字符串、占位符（"..."）、不够长都会触发重试。

🎯
这些模式在任何 AI 应用里都有用

预取（prefetch）+ Promise 复用、三层兜底（主模型→备用模型→mock）、内容校验器——这三个模式是所有生产级 AI 应用的标配。你现在已经在自己的代码里用过一遍了。
