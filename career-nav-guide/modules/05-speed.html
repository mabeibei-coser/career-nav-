让报告提前 3 秒出来

代码里藏着的速度魔法——你的 AI 应用也能用

你以为报告是提交按钮按下才开始生成的？

不是的。代码里有三处「提前启动」，让用户感觉报告「一下子就出来了」。

这就像餐厅在你坐下来翻菜单的时候，厨房就已经在备料了——等你点完单，料已经准备好，上菜明显更快。

🚀
Layer 1：点「下一步」时就预触发

首页点提交 → startReportPrefetch(payload) 和 startQuizPrefetch(payload) 同时被调用 → 量表题目在后台预加载

🚀
Layer 2：Q2 答完立刻启动报告生成

用户答完访谈第 2 题 → startAfterQ2() 被调用 → AI 报告生成在后台开始，此时用户还在答 Q3/Q4

⏳
Layer 3：Loading 页消费已完成的 Promise

用户到达 loading 页时，报告可能已经生成一半甚至全部完成，直接消费结果，不需要重新等待

后台生成是怎么实现的？

Promise 是关键。代码在 Q2 答完时就发出了 AI 请求，并把这个 Promise 存在内存里。到了 loading 页，直接 await 这个已经在跑的 Promise 就行了。

代码翻译：Loading 页是怎么「取货」的

lib/report-client.ts — consumeAll 

const bgPromise = consumeBgGeneratePromise(
  formData, quizAnswers, interviewQ1Q2
);
if (bgPromise !== null) {
  sections = (await bgPromise) as GenerateSections;
}
if (!sections) {
  sections = await callGenerateWithRetry(callPayload, 1);
}
白话翻译

去「bg-runner 仓库」取之前存的那个 Promise（报告生成任务）

把用户数据作为「取货码」，确认是同一次会话的 Promise

取货结束

如果真的取到了 Promise（后台任务在跑或已完成）

等待它完成，拿到报告数据（可能只等很短时间）

结束

如果没取到 Promise（缓存过期或跳过了后台预热）

现场发起报告生成请求，最多重试 1 次（这是最慢的路径）

结束

再看几个你代码里的「工程师思维」

除了速度优化，代码里还有一些值得学习的设计决策：

🛡️

Prompt 注入防御

简历内容用 <resume> 标签包裹，防止用户在简历里写「忽略上述指令」来操控 AI 输出。

🎭

Mock 测试模式

设置 E2E_MOCK_MODE=true，所有 AI 调用返回预设数据——不花 API 费用就能测试完整流程。

🔑

幂等 UUID

每次填表生成唯一 ID，防止用户刷新报告页导致数据库出现重复记录。

⚡

内容校验器

AI 生成的每个字段都有 validator 检查——空字符串、占位符（"..."）、不够长都会触发重试。

🎯
这些模式在任何 AI 应用里都有用

预取（prefetch）+ Promise 复用、三层兜底（主模型→备用模型→mock）、内容校验器——这三个模式是所有生产级 AI 应用的标配。你现在已经在自己的代码里用过一遍了。
